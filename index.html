What this doesn't account for
Factors that may increase real-world GPU count
▼
CORRECTNESSActivation memory during prefill
Up to 10–20GB at 64K+ context. Activations are not held long-term but spike during prefill computation.
→ Add 10–15% VRAM buffer for long-context workloads. Use chunked prefill in vLLM.
ACCURACYLow-concurrency GPU utilisation
At fewer than 20 users, GPU is 20–40% utilised. You may need fewer GPUs in practice.
→ Measure actual utilisation with vLLM /metrics endpoint before scaling.
ADVANCEDSpeculative decoding
A draft model adds VRAM pressure but gives 2–3× throughput. Not modelled here.
→ Add draft model weight memory (typically 1–7B) to your VRAM budget.
ADVANCEDLoRA multi-adapter serving
Each active LoRA adapter adds 16–256MB depending on rank and target modules.
→ Use S-LoRA / punica kernel in vLLM for batched multi-adapter inference.
ADVANCEDPipeline parallelism overhead
Cross-node communication reduces effective throughput 10–30% on Ethernet vs NVLink.
→ Prefer tensor parallelism within a node (NVLink) before pipeline parallelism across nodes.
ƒ Formulas & validation
Core equations and theoretical vs reported actuals
▼
Core equations
1. KV cache per user (exact)
KV_bytes = 2 × num_kv_heads × head_dim × num_layers × bytes_per_element × ctx_tokens
// 2 = K tensor + V tensor  |  bytes: FP16=2, FP8=1, INT8=1
Example Llama 3.1 8B FP16 8K: 2×8×128×32×2×8000 = 1.05 GB/user
2. KV pool available
KV_pool = (GPU_VRAM × TP − model_weights) × (1 − overhead%) × 0.95
// overhead% covers CUDA graphs, activations, NCCL buffers
// 0.95 = PagedAttention block efficiency
3. Users per replica & GPU count
users_per_replica = floor(KV_pool / KV_per_user)
replicas = ceil(concurrent_users / users_per_replica)
total_GPUs = replicas × TP_degree
// TP selected to minimise total_GPUs across TP=1,2,4,8
4. Weight memory
weight_GB = params_B × bytes_per_param  (BF16=2, FP8=1)
// Example: 70B × 2 = 140 GB  |  70B FP8 = 70 GB
